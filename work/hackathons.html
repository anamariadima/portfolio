Evolution: before & after

üìú What we had before? Using the Tokens Studio plugin, our design team could define design tokens (like colors, spacing, or typography) directly in Figma, then devs manually export the JSON file and drop it into the repo.

üîß Why we needed to improve this experience? Before variables launched in Figma in July 2023, teams often asked where to find the latest design tokens. It took some time to learn and test with variables and we also needed a more reliable and transparent way to share token updates.

üíé What we managed to do and what was the impact? Using Token Studio‚Äôs GitHub remote sync, we pushed and pulled tokens directly between Figma and the GitHub repo. This setup made collaboration smoother and kept design and code perfectly aligned.

3/7

Tokens

We have three tiers of tokens: global, semantic and component tokens.

‚Üí Tier 1, the "global" ones, is the foundation of the design system, here we have all the design decisions (all the palettes for colors, sizing, shadows, borders, spacing etc.).

‚Üí In tier 2 we have the semantic tokens that typically reference only the primitive tokens but include guidance on how colors should be used in text, the types of text to use, etc., embedding both meaning and guidance within.

‚Üí And finally, in tier 3 are the component - tokens that are specific to individual components and generally refer to semantic tokens. Component-tokens are valuable for theming scenarios where there is a need to alter not only primary colors but component-specific attributes that really empower themes to capture their unique look and feel.

The example bellow is from our serious JSON file: we have a design decision - a hex value #D23531 named ‚Äúerror.1000‚Äù that is our tier 1 token. Based on our tier 1 we build the reference token named ‚Äúsemantic.error‚Äù - this reference token (ref) will be used in creating background/text/icon/border/shadow/etc. semantic tokens.

Note: It is build in this way because if we want to change the color in the semantic token for error, for example, we‚Äôll change only the ref, and everything else that is linked with this specific ref token will update.

Design and development

With this remote sync, you can easily create automatic pull requests in a GitHub branch and let the dev team review them, or simply push the changes to the main file if a review is not necessary (though, we never push changes directly to the main file, so we need to respect the rules, folks). Additionally, the dev team can make changes directly in the code and designers can ‚Äúpull from GitHub‚Äù to keep the tokens in sync at all time. Feels like a Christmas miracle to me!
